## Script Writing and Execution Event Requirements

### Background
The ollm skills system allows LLMs to write and execute scripts during inference. When an LLM determines it needs to execute a script, it sends a signal back to ollm requesting script execution. This capability is provided through the skills framework where LLMs can generate and execute code dynamically.

### Event Trigger
The script writing and execution event should be captured when:
- An LLM generates a script during inference
- The LLM signals ollm to execute the script
- The script execution is initiated by ollm

Note: From ollm's perspective, the writing and execution appear as a single action, which is acceptable for event tracking purposes.

### Required Event Data
The Script Usage custom event should capture the following information:
- **Script Content**: The complete script code that was generated by the LLM
- **Script Language**: The programming language/type of the script (e.g., Python, bash, etc.)
- **Execution Duration**: Total time taken from script generation signal to execution completion
- **Execution Status**: Success/failure status of the script execution
- **Error Information**: If the script failed, capture error messages and exit codes
- **Session ID**: Correlation ID to link this event to the overall ollm run session
- **Model Name**: The LLM model that generated the script (if known)
- **Timestamp**: When the script writing/execution event occurred
- **Skill Context**: If the script was generated as part of a specific skill, capture the skill name

### Security and Privacy Considerations
- **Script Content Sanitization**: Before logging the script content, ensure it doesn't contain sensitive information such as:
  - API keys or credentials
  - Personal identifiable information (PII)
  - Protected health information (PHI)
  - File paths that might reveal system information
- **Content Filtering**: Implement filtering to redact or hash sensitive patterns in script content
- **Size Limits**: Limit script content size in events to prevent excessive data transmission (e.g., truncate after 10KB)

### Event Schema
```json
{
  "eventType": "ScriptExecution",
  "sessionId": "unique-session-id",
  "modelName": "model-name",
  "scriptLanguage": "python|bash|etc",
  "scriptContent": "sanitized-script-content",
  "executionDurationMs": 1234,
  "executionStatus": "success|failure",
  "errorMessage": "error-details-if-failed",
  "exitCode": 0,
  "skillContext": "skill-name-if-applicable",
  "timestamp": "2026-04-26T12:34:56Z"
}
```

### Implementation Notes
- The event should be triggered at the point where ollm receives the script execution signal from the LLM
- Capture the script content before execution to ensure we have it even if execution fails
- Use asynchronous event logging to avoid impacting script execution performance
- Ensure proper error handling so that event logging failures don't break script execution

## Tool Call Event Requirements

### Background
OLLM integrates with various tools through the MCP (Model Context Protocol) framework. When an LLM determines it needs to use a tool, it makes a tool call request that OLLM processes. These tool calls can be for file operations, API calls, data processing, or other external operations.

### Event Trigger
The tool call event should be captured when:
- An LLM requests a tool call during inference
- OLLM processes and executes the tool call
- The tool call completes (successfully or with failure)

### Required Event Data
The Tool Call custom event should capture the following information:
- **Tool Name**: The name/identifier of the tool that was called
- **Execution Duration**: Total time taken from tool call initiation to completion
- **Execution Status**: Success/failure status of the tool call
- **Error Information**: If the tool call failed, capture error messages, HTTP status codes, and error types
- **Session ID**: Correlation ID to link this event to the overall ollm run session
- **Model Name**: The LLM model that requested the tool call (if known)
- **Timestamp**: When the tool call event occurred

### Security and Privacy Considerations
- **No Parameter Logging**: Tool call parameters must NOT be logged as they may contain PII, PHI, or sensitive information
- **Error Message Sanitization**: Ensure error messages don't expose sensitive information like file paths, credentials, or personal data

### Event Schema
```json
{
  "eventType": "ToolCall",
  "sessionId": "unique-session-id",
  "modelName": "model-name",
  "toolName": "tool-identifier",
  "executionDurationMs": 1234,
  "executionStatus": "success|failure",
  "errorMessage": "sanitized-error-details-if-failed",
  "errorType": "timeout|http_error|exception|etc",
  "httpStatusCode": 500,
  "timestamp": "2026-04-26T12:34:56Z"
}
```

### Implementation Notes
- Event should be triggered at the MCP tool execution layer
- Capture timing from tool call initiation to completion
- Use asynchronous event logging to avoid impacting tool call performance
- Ensure tool call failures don't prevent event logging

## Skills Usage Event Requirements

### Event Trigger
The skills usage event should be captured when:
- An LLM requests to use a specific skill
- OLLM loads and executes the skill
- The skill execution completes (successfully or with failure)

### Required Event Data

- **Skill Name**: The name/identifier of the skill that was used
- **Execution Duration**: Total time taken from skill invocation to completion

- **Skill Context**: Additional context about how the skill was used

### Security and Privacy Considerations

- **Output Filtering**: Skill outputs should not be logged as they may contain processed sensitive data

## Error Event Requirements

### Background
OLLM encounters various types of errors during operation, including configuration errors, model communication failures, tool call failures, and system errors. Tracking these errors is crucial for debugging and system reliability monitoring.

### Event Trigger
The error event should be captured when:
- Any unhandled exception occurs in OLLM
- Expected errors occur (e.g., model unavailable, configuration issues)
- Tool calls or skills fail with errors
- System-level errors are encountered

### Required Event Data
The Error custom event should capture the following information:
- **Error Type**: Classification of the error (system, configuration, model, tool, skill, etc.)
- **Error Message**: The error message (sanitized for sensitive information)
- **Error Source**: Where in the system the error occurred
- **Stack Trace**: Exception stack trace (truncated if necessary)
- **Session ID**: Correlation ID to link this event to the overall ollm run session
- **Model Name**: The LLM model being used when error occurred (if known)
- **Timestamp**: When the error event occurred
- **Context Information**: Additional diagnostic information

### Security and Privacy Considerations
- **Message Sanitization**: Remove file paths, credentials, and personal information from error messages
- **Stack Trace Filtering**: Ensure stack traces don't expose sensitive system information
- **Size Limits**: Truncate large error messages and stack traces to prevent excessive data transmission

### Event Schema
```json
{
  "eventType": "Error",
  "sessionId": "unique-session-id",
  "modelName": "model-name",
  "errorType": "system|configuration|model|tool|skill|network",
  "errorMessage": "sanitized-error-message",
  "errorSource": "module.function|component-name",
  "stackTrace": "sanitized-stack-trace",
  "contextInfo": "additional-diagnostic-info",
  "severity": "low|medium|high|critical",
  "timestamp": "2026-04-26T12:34:56Z"
}
```

### Implementation Notes
- Event should be triggered in exception handlers and error handling code
- Use a global exception handler to catch unhandled exceptions
- Ensure error event logging doesn't cause additional errors
- Sanitize all error information before logging

## Inference Event Requirements

### Background
When OLLM communicates with Ollama for LLM inference, Ollama returns comprehensive metrics about the inference process including timing, token counts, and model information. These metrics are valuable for performance monitoring and optimization.

### Event Trigger
The inference event should be captured when:
- OLLM receives a response from Ollama after an inference request
- The response contains metrics data
- The inference completes (successfully or with failure)

### Required Event Data
The Inference custom event should capture the following information from Ollama's response:
- **Model Name**: The model used for inference
- **Total Duration**: Complete request duration in nanoseconds
- **Load Duration**: Model loading time in nanoseconds
- **Prompt Evaluation**: Token count and duration for prompt processing
- **Response Generation**: Token count and duration for response generation
- **Done Status**: Whether the inference completed successfully
- **Done Reason**: Reason for completion (stop, length, etc.)
- **Session ID**: Correlation ID to link this event to the overall ollm run session
- **Timestamp**: When the inference event occurred

### Security and Privacy Considerations
- **Response Content**: The actual response text should NOT be logged to avoid exposing generated content that might contain sensitive information
- **Prompt Content**: The input prompt should NOT be logged for the same privacy reasons

### Event Schema
```json
{
  "eventType": "Inference",
  "sessionId": "unique-session-id",
  "modelName": "model-name",
  "totalDurationNs": 174560334,
  "loadDurationNs": 101397084,
  "promptEvalCount": 11,
  "promptEvalDurationNs": 13074791,
  "evalCount": 18,
  "evalDurationNs": 52479709,
  "doneStatus": true,
  "doneReason": "stop|length|error",
  "createdAt": "2025-10-17T23:14:07.414671Z",
  "timestamp": "2026-04-26T12:34:56Z"
}
```

### Implementation Notes
- Event should be triggered when processing Ollama responses
- Map Ollama's response fields directly to event fields
- Use asynchronous event logging to avoid impacting inference performance
- Handle cases where Ollama response might be incomplete or malformed
